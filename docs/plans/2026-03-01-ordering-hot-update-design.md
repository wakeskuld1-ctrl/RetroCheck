# 顺序规则热更新设计

## 目标
- 在网关侧引入“顺序域 + 阶段”调度模型，保证跨设备依赖顺序。
- 默认并行，不影响无依赖场景的吞吐。
- 支持管理接口热更新规则，变更不影响已入队任务。

## 约束与范围
- 顺序约束只在网关侧生效，核心层不做二次重排。
- 规则由应用侧维护，通过管理接口动态下发。
- 规则匹配必须走轻量内存路径，避免成为瓶颈。

## 总体架构
- 规则管理层：提供管理接口，支持规则增删改、版本号与原子切换。
- 规则存储层：内存主存 + 可选快照持久化，支持快速回滚。
- 执行层：请求入站后匹配顺序域与阶段，进入对应顺序队列与批处理器。

## 数据流与调度
1. 请求入站解析 device_id 与 msg_type。
2. 规则引擎按 device_prefix + 可选 msg_type 匹配得到 `order_group` 与 `stage`（必要时按 priority 决定优先级）。
3. 进入顺序域队列：
   - 同一顺序域内，阶段严格有序推进。
   - 同阶段内可并行执行。
4. 批处理融合：
   - 批处理仅在同阶段内合并，避免跨阶段乱序。

## 错误处理与一致性
- 单任务失败仅影响自身，不阻塞同阶段其它任务。
- 批处理失败仅影响该批次回执，不回退阶段。
- 队列满时拒绝入队并返回明确错误。
- 热更新采用新旧规则并存 + 原子切换，切换只影响新请求。

## 配置与指标
- 顺序域队列容量、阶段并行度、规则版本号、热更新次数等新增指标。
- 维持现有批处理统计，补充顺序域级指标以定位瓶颈。

## 测试策略
- 顺序域正确性：3 设备 + 3 阶段，验证阶段屏障。
- 默认并行性：不同顺序域并发提交，验证互不阻塞。
- 热更新一致性：规则热切换仅影响新请求。
- 错误与超时：批写失败与超时不回退阶段。
- 5000 设备压力：高并发下顺序不乱且不超时。
